TT;DR: Establish the smallest production-shaped SD-300 native GUI slice without changing the current CLI/TUI. Finish with a pinned, automated CPU/memory GUI packaged into a Corporate MSI that is installed, launched, inspected, and removed on the Alienware host.

## Why
This is a direct operator order and the risk-reduction gate for the larger v3 milestone. The highest-risk seams are Rust/Zig ABI ownership, Native SDK distribution pinning, Windows packaging, and avoiding regressions in the mature v2.0.6 terminal lifecycle. A real MSI slice proves those seams before full UI investment.

Settled decisions: Native SDK 0.5.4 and Zig 0.16.0; compiled native markup with no WebView; Rust `rlib` for the unchanged CLI/TUI plus `cdylib` for the GUI; exact npm tarball and Zig content hash rather than generated absolute `.path`; dark QubeTX design; GUI remains dormant until launched; the app and tray are one process on Windows/macOS; Linux exits on close because the SDK has no Linux tray; Corporate MSI is the first physical installer checkpoint.

## Plan
1. Capture current and immutable v2.0.6 command/output fixtures and record TUI timing/key invariants.
2. Scaffold an owned-build native-rendered app with exact dependency pins and a build-graph audit that rejects local paths.
3. Add a panic-contained, caller-buffer C ABI whose GUI runtime owns its own `SystemSnapshot` and publishes bounded latest-only topics.
4. Implement the first native Overview surface with live CPU/memory values, honest availability, compiled markup, automation, and profiling.
5. Extend the Corporate MSI definition with the GUI, engine, shortcut, and uninstall ownership; install and exercise it on the current Windows host, then remove it and verify no residue.
6. Record the precise hosted feasibility work still required for macOS and all Linux targets before promoting the full-GUI task.

## Impact
Intended: creates the reusable engine/app/package foundation while leaving the TUI control flow and existing public commands unchanged. Possible unintended impact: Cargo crate type changes, Windows DLL search mistakes, ABI mismatch, new build requirements, installer file-table/upgrade conflicts, or CLI snapshot drift. Guard with isolated modules, absolute app-relative loading, exact version checks, golden fixtures, and a real install/uninstall smoke.

## Acceptance
The branch contains a reproducible Native SDK app and Rust engine that build together, render live SD-300 CPU/memory data, pass headless and live automation, and install through a Corporate MSI without changing normal CLI/TUI behavior. The MSI can be removed through the supported owner path with no GUI residue.
